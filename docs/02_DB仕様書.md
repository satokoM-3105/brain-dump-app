# ブレインダンプアプリ — DB仕様書（Supabase / PostgreSQL）

## 1. 概要

本アプリの永続化は **Supabase（PostgreSQL）** で行う。

| テーブル名 | 役割 |
|------------|------|
| `brain_dumps` | メモ本体（タイトル・本文・作成日時・カテゴリ参照） |
| `categories` | カテゴリの**マスタ**（ドロップダウン選択肢の元データ） |

要求仕様 **FR-07（カテゴリ管理）** を満たすため、`brain_dumps` に任意の `category_id` を持たせ、`categories` で選択肢を管理する。

要求仕様 **FR-08（ユーザー別メモ管理）** を満たすため、`brain_dumps` に **`user_id` カラム** を追加し、メモの所有者を Supabase Auth のユーザと紐づける。

---

## 2. ER 構成（論理）

```
categories（マスタ）
     ↑
     │ 0..1（category_id、NULL 可）
     │
brain_dumps（メモ）

auth.users ── 1:N ── brain_dumps   ※認証導入時（user_id）
```

| 関係 | 説明 |
|------|------|
| `categories` → `brain_dumps` | 1 カテゴリに複数メモが紐づきうる（1 : N） |
| `auth.users` → `brain_dumps` | 1 ユーザに複数メモ（1 : N）。`brain_dumps.user_id` で所有者を表す |
| `brain_dumps.category_id` | **NULL 可**。未選択のメモはカテゴリなし（画面上は「未分類」等） |
| `categories` 削除時 | `brain_dumps.category_id` は `ON DELETE SET NULL` とし、メモは残す |

## 3. テーブル定義：`categories`（カテゴリマスタ）

要求仕様 FR-07 に基づき、**先に本テーブルを作成**してから `brain_dumps.category_id` の外部キーを張る。

### 3.1 カラム

| カラム名 | 型 | 制約 | 説明 |
|----------|-----|------|------|
| `id` | `uuid` | `PRIMARY KEY`、`DEFAULT gen_random_uuid()` | カテゴリの識別子 |
| `name` | `text` | `NOT NULL`、`UNIQUE` | カテゴリ名（ドロップダウン・一覧の表示名） |
| `created_at` | `timestamptz` | `NOT NULL`、`DEFAULT now()` | 登録日時 |

### 3.2 運用方針

- 選択肢は**必ず本テーブルの行**で管理する（アプリ内の固定配列にしない）。
- マスタの追加・変更・削除を**画面から行う機能**は要求仕様の対象外。初期・追加データは SQL または Supabase Dashboard で投入する。
- 参照用マスタのため、現行フェーズでは **RLS を有効にしない**想定（`anon` からの `SELECT` でドロップダウンを満たす）。

### 3.3 物理DDL

```sql
create table public.categories (
  id uuid primary key default gen_random_uuid(),
  name text not null unique,
  created_at timestamptz not null default now()
);

comment on table public.categories is 'メモ用カテゴリのマスタ';
```

## 4. テーブル定義：`brain_dumps`（メモ）

### 4.1 カラム

| カラム名 | 型 | 制約 | 説明 |
|----------|-----|------|------|
| `id` | `uuid` | `PRIMARY KEY`、`DEFAULT gen_random_uuid()` | レコードの識別子 |
| `user_id` | `uuid` | `NOT NULL`、`REFERENCES auth.users(id) ON DELETE CASCADE` | メモの所有者。ログインユーザと紐づける（FR-08） |
| `title` | `text` | `NOT NULL` | タイトル（必須） |
| `content` | `text` | `NOT NULL` | 本文（必須） |
| `category_id` | `uuid` | **`NULL` 可**、`REFERENCES categories(id) ON DELETE SET NULL` | カテゴリ（**任意**。FR-07：必須ではない） |
| `created_at` | `timestamptz` | `NOT NULL`、`DEFAULT now()` | 作成日時（自動セット） |

- `category_id` が `NULL` の行は、カテゴリ未設定のメモとして扱う。
- **`user_id` は必須**。未ログイン状態ではメモ行を作成しない（アプリ側でログイン必須とする）。
- 一覧表示時は `categories` と結合（または別クエリ）し、名前を表示する。
- **参照・追加の範囲**: 各ユーザは `user_id = auth.uid()` に一致する行のみを対象とする（アプリ・RLS 等で担保。詳細は機能仕様書）。

### 4.2 ユーザー別メモ管理に伴う DB 変更（FR-08）

**本要件における DB スキーマ変更は、`brain_dumps` への `user_id` カラム追加のみ**とする。トリガ・RLS・インデックスの追加は本変更の対象外（必要に応じて別途検討）。

#### 変更内容

| 項目 | 内容 |
|------|------|
| 対象テーブル | `brain_dumps` のみ |
| 追加カラム | `user_id` |
| 型 | `uuid` |
| 外部キー | `auth.users(id) ON DELETE CASCADE` |
| NULL | **NOT NULL**（既存行がある場合は移行方針を別途決める） |

#### 適用 SQL（列追加）

`brain_dumps` が既に存在する環境では、次の **列追加** で対応する。

```sql
alter table public.brain_dumps
  add column user_id uuid references auth.users (id) on delete cascade;

comment on column public.brain_dumps.user_id is 'メモの所有者（Supabase Auth ユーザ ID）';
```

既存データに `user_id` が NULL の行がある場合、`NOT NULL` 制約を付ける前に所有者の割当または削除が必要である。

#### データの意味

| 操作 | `user_id` の扱い |
|------|------------------|
| INSERT（新規メモ） | ログイン中ユーザの ID をセットする |
| SELECT（一覧・参照） | ログイン中ユーザの ID と一致する行のみ対象 |
| UPDATE / DELETE | 同上（他ユーザの行は対象外） |

`user_id` のセット方法（アプリから送る、DB トリガ、RLS 等）は **機能仕様書・アプリ実装** で定める。DB 変更としては列の存在と外部キー制約までを本書で定義する。

### 4.3 既存テーブルへの変更（カテゴリ追加）

`brain_dumps` が既に存在する環境では、新規作成ではなく **列追加** で対応する。

```sql
alter table public.brain_dumps
  add column category_id uuid references public.categories (id) on delete set null;

comment on column public.brain_dumps.category_id is 'カテゴリ（任意。未設定時は NULL）';
```

### 4.4 物理DDL（新規環境で両方まとめて作る場合の例）

```sql
-- 1. マスタ
create table public.categories (
  id uuid primary key default gen_random_uuid(),
  name text not null unique,
  created_at timestamptz not null default now()
);

-- 2. メモ（user_id あり）
create table public.brain_dumps (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users (id) on delete cascade,
  title text not null,
  content text not null,
  category_id uuid references public.categories (id) on delete set null,
  created_at timestamptz not null default now()
);

comment on table public.brain_dumps is 'ブレインダンプ用メモ';
comment on column public.brain_dumps.category_id is 'カテゴリ（任意。未設定時は NULL）';
```

## 5. インデックス方針

| 名前 | テーブル | 定義 | 目的 |
|------|----------|------|------|
| `brain_dumps_created_at_idx` | `brain_dumps` | `(created_at DESC)` | 一覧の新しい順取得（補助） |
| `brain_dumps_category_id_idx` | `brain_dumps` | `(category_id)` | カテゴリ結合・絞り込みの効率化 |
| `brain_dumps_user_id_created_at_idx` | `brain_dumps` | `(user_id, created_at DESC)` | ユーザ別一覧取得（推奨・本 DB 変更の必須要件ではない） |

```sql
create index brain_dumps_created_at_idx
  on public.brain_dumps (created_at desc);

create index brain_dumps_category_id_idx
  on public.brain_dumps (category_id);
```

## 8. セキュリティ方針（要約）

| 対象 | 方針 |
|------|------|
| `brain_dumps` | `user_id` により所有者を識別。RLS で `user_id = auth.uid()` の行のみ CRUD |
| `categories` | 共通マスタ。RLS で SELECT のみ許可（INSERT / UPDATE / DELETE は拒否） |

Row Level Security の **ポリシー詳細** は **`docs/05_RLS仕様書.md`** を参照。SQL の生成・適用は別途行う。

---

## 9. トリガ・ストレージ

- ストレージ（画像・ファイル）の利用要件はない。
- `created_at` は `DEFAULT now()` で足りるため、専用トリガは必須としない。
