# ブレインダンプアプリ — 機能仕様書

## 1. 概要

| 項目 | 内容 |
|------|------|
| アプリ名 | ブレインダンプ |
| 目的 | 思考・アイデアをテキストで保存し、一覧・編集・削除できるようにする |
| 実装 | HTML / CSS / JavaScript（`index.html`）、Supabase JavaScript SDK（CDN） |
| データ | Supabase の `brain_dumps`・`categories` テーブル |
| 認証 | **Supabase Auth**（メール＋パスワード）。新規登録・ログイン・ログアウト・メール表示 |
| セキュリティ | **`brain_dumps.user_id`** によりユーザごとにメモを分離（FR-08）。参照・追加は自分のメモのみ |

本書は、**画面上の操作がどの処理になるか**を機能単位で定義する。実装者が JavaScript の処理を書く際の設計書とする。

テーブル構造は **`docs/02_DB仕様書.md`** を参照。`brain_dumps` は `id`, **`user_id`**, `title`, `content`, `category_id`, `created_at` を使用。`categories` は `id`, `name` をドロップダウン用に使用する。

**FR-08（ユーザー別メモ管理）**: DB 変更は `brain_dumps.user_id` 列の追加のみ。アプリはログイン後、**自分の `user_id` のメモのみ**を参照・追加する（本書 F-12 および各メモ機能で定義）。

**Supabase プロジェクト設定**: Authentication → Providers → Email の **Confirm email を無効** にすること（メール確認なしで即ログイン可能にする）。

## 3. 共通仕様

### 3.1 Supabase クライアント

- ページ読み込み時に、CDN から読み込んだ Supabase SDK でクライアントを初期化する
- 接続情報（プロジェクト URL・anon key）は **環境に応じて設定**する（本仕様書には具体的な値を書かない）
- 認証付きリクエストでは、ログイン後のセッション JWT が自動的に付与される

### 3.2 認証共通

| 項目 | 仕様 |
|------|------|
| 方式 | メール＋パスワードのみ |
| パスワード最小長 | 6 文字（クライアントで事前チェック） |
| メール確認 | 行わない（Supabase 側で Confirm email 無効） |
| 認証エラー | `alert()` で表示 |
| 登録・ログイン成功後 | `index.html` 上でメモ機能エリアを表示 |
| ログアウト後 | `index.html` 上で認証エリアを表示 |

### 3.3 ユーザー別メモ管理（FR-08 / F-12）

| 項目 | 仕様 |
|------|------|
| 所有者カラム | `brain_dumps.user_id`（`auth.users.id` への外部キー） |
| 参照 | ログイン中ユーザの ID と `user_id` が一致する行**のみ**一覧・取得する |
| 追加 | 新規 INSERT 時、行に **ログイン中ユーザの `user_id`** を付与する |
| 編集・削除 | 参照と同様、自分の `user_id` の行のみ対象（F-03・F-04） |
| 他ユーザのメモ | 一覧に表示しない。追加・更新・削除も不可 |
| `user_id` の送信 | INSERT 時にクライアントから `{ title, content, category_id, user_id }` を送る、または DB 側で `auth.uid()` をセットする等、実装でいずれかを採用する（DB 変更は列追加のみのため、セット方法はアプリ実装で決定） |
| 未ログイン | メモの参照・追加は行わない（F-11） |

### 3.4 対象テーブル

| テーブル名 | 用途 |
|------------|------|
| `brain_dumps` | メモの保存・一覧・更新・削除 |
| `categories` | カテゴリ選択肢のマスタ（ドロップダウン・一覧表示名） |

### 3.5 バリデーション（入力チェック）

| チェック内容 | 条件 | 失敗時 |
|--------------|------|--------|
| パスワード長（認証） | 6 文字以上 | `alert()` で表示、Auth API は呼ばない |
| タイトル必須 | 前後の空白を除いて1文字以上 | F-05 で入力エラーを表示、DB操作は行わない |
| 本文必須 | 前後の空白を除いて1文字以上 | 同上 |
| カテゴリ | **必須チェックなし** | 未選択（空 value）のときは `category_id` を `null` として保存・更新 |

### 3.6 編集モードの管理（内部状態）

JavaScript 側で次の変数（名前は実装自由）を持つ想定。

| 変数の意味 | 新規作成時 | 編集時 |
|------------|------------|--------|
| 編集中のメモ ID | `null` または未設定 | 対象メモの `id`（uuid） |

- 編集中 ID があるとき、保存ボタンは **更新（update）** として動く
- 編集中 ID がないとき、保存ボタンは **新規（insert）** として動く

## 4. 各機能の詳細

### F-07 新規登録機能

#### 目的

メールアドレスとパスワードで Supabase Auth にユーザーを登録する。

#### 処理の流れ

| 順番 | 処理内容 |
|------|----------|
| 1 | ユーザーがメール・パスワードを入力する |
| 2 | 新規登録ボタンを押す |
| 3 | メール空欄・パスワード 6 文字未満をチェック。NG なら `alert()` |
| 4 | `auth.signUp({ email, password })` を実行 |
| 5 | 成功 → 同一 `index.html` 内でメモ機能を表示 |
| 6 | 失敗 → `alert(error.message)` |

#### Supabase 操作

- **操作**: `auth.signUp`

---

### F-08 ログイン機能

#### 目的

登録済みメールアドレスとパスワードでセッションを取得する。

#### 処理の流れ

|------|----------|
| 1 | ユーザーがメール・パスワードを入力する |
| 2 | ログインボタンを押す |
| 3 | 入力チェック（F-07 と同様） |
| 4 | `auth.signInWithPassword({ email, password })` を実行 |
| 5 | 成功 → 同一 `index.html` 内でメモ機能を表示 |
| 6 | 失敗 → `alert(error.message)` |

#### Supabase 操作

- **操作**: `auth.signInWithPassword`

### F-11 認証ガード機能

#### 目的

|------|----------|
| 1 | `index.html` 読み込み時に `auth.getSession()` |

#### Supabase 操作

- **操作**: `auth.getSession`

### F-12 ユーザー別メモ管理機能

#### 目的

要求仕様 **FR-08** に基づき、**ユーザーはログイン後、自分が作成したメモのみを参照・追加できる**ようにする。

#### 処理の流れ

| 順番 | 処理内容 |
|------|----------|
| 1 | ログイン成功後、セッションからユーザ ID（`auth.uid()` 相当）を取得する |
| 2 | **参照（F-02）**: `brain_dumps` を取得する際、結果が **自分の `user_id` の行のみ** となるようにする |
| 3 | **追加（F-01）**: INSERT する行に **ログイン中ユーザの `user_id`** をセットする |
| 4 | 他ユーザの `user_id` を持つ行は、一覧・編集・削除の対象に含めない |

#### データの扱い

| 操作 | `user_id` |
|------|-----------|
| SELECT（一覧） | 自分の ID と一致する行のみ返ること |
| INSERT | 自分の ID が付与された行として作成されること |
| UPDATE / DELETE | 自分の ID の行のみ成功すること（他ユーザ行は対象外） |

#### 正常時の動き

- ユーザ A でログイン → A が作成したメモのみ一覧表示・新規追加可能
- ユーザ B でログイン → B のメモのみ表示。A のメモは見えない

#### エラー時の動き

| エラー種別 | 動き |
|------------|------|
| 未ログインでメモ操作 | F-11 により認証エリア表示。DB 操作は行わない |
| 他ユーザ行への操作 | 一覧に含まれない、または DB/API が拒否（実装・RLS 次第） |

#### Supabase 操作

- **操作**: `select` / `insert`（および F-03・F-04 の `update` / `delete`）
- **テーブル**: `brain_dumps`
- **条件**: `user_id` = ログイン中ユーザの ID

### F-01 メモ新規保存機能

#### 目的

ユーザーが入力したタイトル・本文・カテゴリ（任意）を、新しいメモとしてデータベースに保存する。

#### 処理の流れ

| 順番 | 処理内容 |
|------|----------|
| 1 | ユーザーがタイトル・本文・カテゴリ（任意）を入力する |
| 2 | 保存ボタンを押す |
| 3 | タイトル・本文の必須チェックを行う |
| 4 | チェック OK かつログイン済みなら `brain_dumps` に **insert** する。**`user_id` にログイン中ユーザの ID をセット**する（F-12） |
| 5 | 成功したらタイトル・本文・カテゴリ選択を初期状態に戻す |
| 6 | 編集中 ID があればクリアする（新規保存モードに戻す） |
| 7 | メモ一覧を再取得して画面を更新する |
| 8 | F-05 で「保存成功」メッセージを表示する |

#### 入力値

| 項目 | 来源 | 必須 |
|------|------|------|
| `title` | タイトル入力欄 | はい |
| `content` | 本文入力欄 | はい |
| `category_id` | カテゴリドロップダウン | いいえ（未選択可） |
| `user_id` | ログインセッション（F-12） | はい（INSERT 時に自動付与） |

※ `id`, `created_at` は DB 側で自動設定（クライアントから送らない想定でよい）。

#### 出力・表示内容

- 一覧に新しいメモが先頭付近（新しい順）に表示される
- メッセージ：「メモを保存しました」など

#### 正常時の動き

- insert が成功
- 入力欄が空になる
- 一覧が最新状態になる
- 成功メッセージが表示される

#### エラー時の動き

| エラー種別 | 動き |
|------------|------|
| 入力エラー（空欄など） | insert しない。F-05 で入力エラー表示。入力内容は残す |
| Supabase エラー | insert しない。F-05 で接続エラーまたはその他エラー表示。入力内容は残す |

#### Supabase 操作

- **操作**: `insert`
- **テーブル**: `brain_dumps`
- **送るデータの例**: `{ title, content, category_id, user_id }`（`user_id` はログイン中ユーザ ID。`category_id` は未選択時 `null`）

### F-02 メモ一覧表示機能

#### 目的

保存済みの**自分の**メモを一覧で見られるようにする。画面を開いたとき、および保存・更新・削除のあとに最新状態を反映する。

#### 処理の流れ

| 順番 | 処理内容 |
|------|----------|
| 1 | 画面（ページ）を開いたとき、または一覧の再表示が必要なとき |
| 2 | `brain_dumps` から **select** する。**自分の `user_id` の行のみ**が対象（F-12）。カテゴリ名表示のため `categories` と結合 |
| 3 | `created_at` の **降順**（新しい順）で並べる |
| 4 | 取得結果をメモ一覧エリアに HTML として描画する |
| 5 | 各メモに編集ボタン・削除ボタンを付ける |

#### 入力値

- なし（一覧取得のみ）

#### 出力・表示内容

各メモについて、少なくとも次を表示する。

| 表示項目 | データ元 |
|----------|----------|
| タイトル | `title` |
| カテゴリ名 | `categories.name`（`category_id` がある場合）。NULL のときは「未分類」等 |
| 作成日時 | `created_at`（表示用に整形） |
| 本文プレビュー | `content`（全文または一部） |
| 編集ボタン | — |
| 削除ボタン | — |

内部では各行に `id` を紐づけて保持する（`data-id` 属性など）。

#### 正常時の動き

- メモが0件のときは「メモはありません」などの文言を一覧エリアに表示してもよい
- 1件以上あるときは新しい順に並べる

#### エラー時の動き

| エラー種別 | 動き |
|------------|------|
| Supabase エラー（取得失敗） | 一覧を空にするか前回表示のままとする（実装方針で決める）。F-05 で接続エラー表示 |

#### Supabase 操作

- **操作**: `select`
- **テーブル**: `brain_dumps`
- **取得カラム**: `id`, **`user_id`**, `title`, `content`, `category_id`, `created_at` および結合した `categories.name`（実装例: `.select('id, user_id, title, content, category_id, created_at, categories(name)')`）
- **絞り込み**: `user_id` = ログイン中ユーザ ID（F-12。`.eq('user_id', uid)` または RLS 等）
- **並び順**: `created_at` 降順

### F-03 メモ編集機能

#### 目的

既存メモのタイトル・本文・カテゴリ（任意）を変更し、データベースに反映する。

#### 処理の流れ

| 順番 | 処理内容 |
|------|----------|
| 1 | 一覧の編集ボタンを押す |
| 2 | そのメモの `id`, `title`, `content`, `category_id` を取得する（一覧描画時に保持したデータを使う） |
| 3 | タイトル入力欄・本文入力欄・カテゴリドロップダウンに値をセットする |
| 4 | JavaScript 内に **編集中のメモ ID** を保存する |
| 5 | ユーザーが内容を直し、保存ボタンを押す |
| 6 | 必須チェックを行う |
| 7 | チェック OK かつ編集中 ID がある場合、`brain_dumps` を **update** する（条件: `id` が一致 **かつ `user_id` が自分**） |
| 8 | 成功したら入力欄を空にし、編集中 ID をクリアする |
| 9 | F-02 で一覧を再表示する |
| 10 | F-05 で「更新成功」メッセージを表示する |

#### 入力値

| 項目 | 来源 | 必須 |
|------|------|------|
| `id` | 編集ボタン押下時に保持した uuid | はい（更新時） |
| `title` | タイトル入力欄 | はい |
| `content` | 本文入力欄 | はい |
| `category_id` | カテゴリドロップダウン | いいえ |

#### 出力・表示内容

- 一覧上の該当メモのタイトル・本文・カテゴリ名・日時が更新後の内容になる（`created_at` は変更しない）
- メッセージ：「メモを更新しました」など

#### 正常時の動き

- update が成功
- 入力欄が空になり、新規作成モードに戻る
- 一覧が再読み込みされる

#### エラー時の動き

| エラー種別 | 動き |
|------------|------|
| 入力エラー | update しない。入力エラー表示 |
| 編集中 ID がないのに保存された | 新規保存（F-01）として扱うか、エラーとする（実装で一貫させる。推奨: 編集中 ID がなければ insert） |
| Supabase エラー | 入力内容・編集中 ID は残す。エラーメッセージ表示 |

#### Supabase 操作

- **操作**: `update`
- **テーブル**: `brain_dumps`
- **条件**: `id` = 編集中のメモ ID **かつ `user_id` = ログイン中ユーザ ID**（F-12）
- **更新する列**: `title`, `content`, `category_id`（未選択時は `null`）

### F-04 メモ削除機能

#### 目的

不要になったメモをデータベースから削除し、一覧から消す。

#### 処理の流れ

| 順番 | 処理内容 |
|------|----------|
| 1 | 一覧の削除ボタンを押す |

| 3 | ユーザーがキャンセルしたら何もしない |
| 4 | OK の場合、対象メモの `id` **かつ自分の `user_id`** で `brain_dumps` から **delete** する |
| 5 | 成功したら F-02 で一覧を再表示する |
| 6 | 削除したメモが編集中だった場合は、入力欄と編集中 ID をクリアする |
| 7 | F-05 で「削除成功」メッセージを表示する |

#### 入力値

| 項目 | 来源 | 必須 |
|------|------|------|
| `id` | 削除ボタンに紐づくメモの uuid | はい |

#### 出力・表示内容

- 一覧から該当行が消える
- メッセージ：「メモを削除しました」など

#### 正常時の動き

- delete が成功
- 一覧が最新状態になる

#### エラー時の動き

| エラー種別 | 動き |
|------------|------|
| ユーザーがキャンセル | DB操作なし |
| Supabase エラー | メモは一覧に残る。F-05 でエラー表示 |

#### Supabase 操作

- **操作**: `delete`
- **テーブル**: `brain_dumps`
- **条件**: `id` = 対象メモの ID **かつ `user_id` = ログイン中ユーザ ID**（F-12）

### F-05 メッセージ表示機能

#### 目的

操作の結果やエラーを、ユーザーが画面上で確認できるようにする。

#### 処理の流れ

| 順番 | 処理内容 |
|------|----------|
| 1 | 他機能（F-01〜F-04）から、表示したいメッセージ種別と文言が決まる |
| 2 | メッセージ表示エリアにテキストをセットする |
| 3 | （任意）成功系は数秒後にメッセージをクリアする |
| 4 | （任意）メッセージ種別に応じて色やクラスを変えて見分けやすくする |

#### 表示するメッセージ種別

| 種別 | いつ表示するか | 表示例 |
|------|----------------|--------|
| 保存成功 | F-01 成功後 | メモを保存しました |
| 更新成功 | F-03 成功後 | メモを更新しました |
| 削除成功 | F-04 成功後 | メモを削除しました |
| 入力エラー | 必須チェック失敗時 | タイトルと本文を入力してください |
| Supabase接続エラー | ネットワーク・認証・テーブル参照など取得・更新系の失敗 | データの取得に失敗しました。しばらくしてから再度お試しください |
| その他エラー | 上記に当てはまらない失敗 | エラーが発生しました |

#### 入力値

- メッセージ種別（内部の区分）
- 表示文言（固定文またはエラー内容の要約）

#### 出力・表示内容

- メッセージ表示エリアに1行〜数行のテキスト

#### 正常時の動き

- 指定した文言が画面に表示される

#### エラー時の動き

- 本機能自体は DB に依存しないため、表示エリアが DOM に存在しない場合のみ実装不備

#### Supabase 操作

- **なし**

### F-06 カテゴリ一覧取得機能

#### 目的

カテゴリマスタから選択肢を取得し、メモ投稿・編集用のドロップダウンを表示する。

#### 処理の流れ

| 順番 | 処理内容 |
|------|----------|
| 1 | ページ読み込み時（メモ一覧と同時でよい） |
| 2 | `categories` から **select** する |
| 3 | 取得結果を `select` 要素の `option` として描画する |
| 4 | 先頭に未選択用の option（value 空、`category_id` = null 相当）を置く |

#### 入力値

- なし

#### 出力・表示内容

| 表示 | データ元 |
|------|----------|
| ドロップダウン各項目 | `categories.id`（value）、`categories.name`（表示文言） |

#### 正常時の動き

- マスタが0件でも、未選択 option のみでドロップダウンを表示できる
- 1件以上あるときは名前順または登録順で並べてよい（実装で統一）

#### エラー時の動き

| エラー種別 | 動き |
|------------|------|
| Supabase エラー | ドロップダウンは未選択のみ、または空。F-05 で接続エラー表示（任意） |

#### Supabase 操作

- **操作**: `select`
- **テーブル**: `categories`
- **取得カラム**: `id`, `name`（最低限）

## 9. カテゴリ実装メモ（FR-07）

| 処理 | 実装の要点 |
|------|------------|
| ドロップダウン生成 | F-06 で `categories` を取得し、`option` を動的生成 |
| 保存・更新 | `category_id`: 未選択 → `null`、選択時 → 選択した uuid |
| 所有者 | INSERT 時に `user_id` = ログイン中ユーザ ID（F-12） |
| 一覧 | `brain_dumps` を **自分の `user_id`** で取得。`categories` と結合して `name` を取得。無い場合は表示「未分類」 |
| 編集読み込み | `cachedMemos` に `category_id` を保持し、ドロップダウンに反映 |
